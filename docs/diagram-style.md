# Article Diagram Style

This project uses a hand-drawn architecture-diagram style for article images.

## Style Name

Use an Excalidraw / Rough.js style:

- Hand-drawn, slightly imperfect outlines.
- Cream paper background.
- Thick black arrows for main flow.
- Coral red for "before / chaos / problem".
- Sage green for "after / clarity / solution".
- Rounded cards, browser-window panels, small labels, and simple icon-like marks.

## Online Implementations

Good implementation references:

- Excalidraw: https://excalidraw.com
- Excalidraw GitHub: https://github.com/excalidraw/excalidraw
- Rough.js: https://roughjs.com
- Rough.js GitHub: https://github.com/rough-stuff/rough
- diagrams.net rough style: https://www.drawio.com/blog/rough-style

For this repo, prefer deterministic SVG output. Use Rough.js-like SVG paths, or manually draw SVGs that follow the same visual grammar.

## Canvas

- Default size: `1200 x 720` or `1600 x 900`.
- Aspect ratio: `16:9`.
- Background: warm cream, such as `#f8f1e3`, `#f7f0df`, or `#f8f5ee`.
- Optional subtle vignette or paper-like warmth is allowed, but avoid glossy gradients.

## Palette

- Ink: `#161412` or `#1f1d1a`.
- Muted text: `#68645d`.
- Problem coral: `#df765d`, `#e27b61`, `#d96f57`.
- Solution sage: `#91bba5`, `#8fbea6`, `#7fb196`.
- Soft panel fill: `#fff8eb`, `#f7efdf`, `#eef5ec`.

Keep the palette restrained. The primary contrast is usually coral vs sage.

## Typography

- Use Chinese-first sans fonts with a friendly hand-drawn feel where available:
  - `LXGW WenKai`
  - `PingFang SC`
  - `Microsoft YaHei`
  - `Arial`
- Titles should be large, bold, and simple.
- Body labels should be short. Prefer phrases over paragraphs.
- Avoid tiny text that becomes unreadable after Yuque scaling.

## Shape Language

- Rounded rectangles with black stroke, `rx` around `8-14`.
- Browser/window panels with three colored dots in the top-left.
- Small pill labels for counts or tags.
- Simple hook/check/icon marks can be drawn with strokes.
- Use slightly irregular or doubled strokes when possible.

## Layout Pattern

Most article diagrams should use one of these structures:

- Before / After: left side chaotic, right side clear.
- Funnel: scattered inputs converge into a structured system.
- Loop: repeated engineering cycle with checkpoints.
- Layered stack: product, system, data, quality, operations.

For "before / after" diagrams:

- Left side: coral, dense, tangled, many crossing arrows.
- Right side: sage, aligned, modular, fewer arrows.
- Center: thick black arrow or transition label.
- Add a numeric simplification if possible, such as `17 -> 6-8`, `20+ -> 3`, or `34 -> 9 groups`.

## Arrows And Lines

- Main arrows: thick black, `stroke-width` around `8-14`, filled arrowheads.
- Secondary arrows: black or muted ink, `stroke-width` around `2-4`.
- Messy state can use overlapping curved lines.
- Clear state should use clean orthogonal or gently curved connectors.

## Article Image Rules

- Every diagram should explain one idea without needing the article paragraph beside it.
- Prefer Chinese labels for Yuque article images.
- Use short, concrete labels: "混乱", "清晰", "任务分组", "状态流转", "评审清单".
- Do not over-decorate. The sketch style is the decoration.
- Export images as SVG when possible. PNG is acceptable when generated by an image model.

## Reusable Prompt

When generating a bitmap image, use this prompt style:

```text
Hand-drawn Excalidraw-style software architecture diagram on a warm cream paper background. Thick rough black outlines and arrows, rounded browser-window panels, coral red "before/chaos" section on the left, sage green "after/clarity" section on the right, Chinese labels, simple icons, slightly imperfect sketch lines, clean 16:9 composition, no photorealism, no glossy gradients.
```

## Checklist

Before publishing an article image:

- Does it have a warm cream paper background?
- Are problem/solution colors coral and sage?
- Are main arrows thick and readable?
- Are labels short and readable in Yuque?
- Does the image communicate the article's core idea by itself?
- Is the output stored under `assets/yuque/` if it is a repo asset?
